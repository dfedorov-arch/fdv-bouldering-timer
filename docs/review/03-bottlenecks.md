# 03. Узкие места и риски

Каждая позиция: где (код), в чём проблема, чем грозит, тяжесть (severity). Тяжесть выставлена с приоритетом на две заявленные цели: (А) **точная синхронизация таймеров/сигналов между браузерами** и (Б) **сеть не должна останавливать таймеры**.

Сводка:

| # | Проблема | Severity | Цель |
| --- | --- | --- | --- |
| B1 | Сервер считает время по настенным часам `Date.now()` | **Critical** | А |
| B2 | Состояние таймера только в памяти, нет восстановления после рестарта | **Critical** | Б |
| B3 | Предел скорости часов ±10 % слишком широкий | **High** | А |
| B4 | Сигнал «молча» пропускается при подвисании у границы сегмента | **High** | А |
| B5 | Дрейф модели часов накапливается во время офлайна | **High** | А+Б |
| B6 | Управляющие действия теряются без обратной связи при сбое | **Medium** | Б |
| B7 | Единая точка отказа (сервер/primary), нет автоперезапуска | **Medium** | Б |
| B8 | Нет аутентификации — любой в LAN может сбросить таймер | **Medium** | — |
| B9 | SSE может «зависнуть» без `onerror`, нет heartbeat-детекции на клиенте | **Medium** | Б |
| B10 | Нет автоматических тестов на логику времени | **Medium** | А |
| B11 | Размер state-пейлоада растёт с числом клиентов, рассылается часто | **Low** | — |
| B12 | Дрейф между часами AudioContext и performance.now() | **Low** | А |
| B13 | Троттлинг фоновой вкладки на дисплеях | **Low** | А |

<a id="b1"></a>
## B1. Сервер считает время по `Date.now()` (Critical, цель А)

**Где.** `serve-bouldering-timer.js:469`

```js
function elapsedSeconds(now = Date.now()) {
  if (!timerState.running) return timerState.elapsedBeforePause;
  return Math.max(0, (now - timerState.startedAt) / 1000);
}
```

`startedAt`, `now`, `serverSentAt`/`serverReceivedAt` — всё на `Date.now()` (настенные часы).

**Проблема.** `Date.now()` подвержен скачкам системного времени: NTP-степ, ручная правка часов, коррекция при выходе ноутбука из сна. Если во время идущего раунда часы сервера прыгнут на ±N секунд:

- Серверный `elapsed` мгновенно изменится на N.
- Это значение через клиентскую модель разойдётся **на все экраны сразу** → неверное оставшееся время и сдвиг всех сигналов **одновременно везде**.
- Из-за клампа коррекции 250 мс/обновление (`index.html:1860`) разные клиенты будут «догонять» скачок с разной скоростью → ещё и **временное расхождение между браузерами**.

Запланированный старт по часам (`scheduledStartTime`) обязан опираться на настенное время — это нормально. Но **длительность идущего раунда должна считаться по монотонным часам**.

**Фикс (суть).** Якорить старт раунда на монотонные часы сервера (`process.hrtime.bigint()` / `performance.now()` из `perf_hooks`) и считать `elapsed` по ним; настенное `Date.now()` оставить только для планового старта и для метки `now` в синхронизации офсета. См. [план, шаг 1.1](04-improvement-plan.md#s11).

<a id="b2"></a>
## B2. Состояние таймера только в памяти (Critical, цель Б)

**Где.** `serve-bouldering-timer.js:251` — `timerState` живёт лишь в процессе. Нет ни persistence, ни восстановления.

**Проблема.** Худший сценарий для «сбой не должен останавливать таймер» — это сбой **сервера**, а не сети. Если процесс Node упадёт, или компьютер-сервер уйдёт в сон/перезагрузится, или лаунчер будет случайно закрыт во время раунда — **идущий отсчёт теряется полностью**, состояние стартует с нуля (`idle`). Клиенты при переподключении получат «таймер не запущен».

Это прямое противоречие ключевому требованию надёжности.

**Фикс (суть).**
1. Персистить `timerState` на диск атомарно (temp-файл + `rename`) при каждом изменении версии.
2. При старте сервера восстанавливать состояние; для идущего раунда пересчитывать `startedAt` от сохранённого абсолютного времени старта.
3. Обернуть процесс авто-рестартом (см. B7).

См. [план, шаг 1.2](04-improvement-plan.md#s12).

<a id="b3"></a>
## B3. Предел скорости часов ±10 % слишком широкий (High, цель А)

**Где.** `index.html:1598`

```js
const syncRateMin = 0.90;
const syncRateMax = 1.10;
```

**Проблема.** Реальный дрейф кварцевых часов устройств — порядка 10–100 ppm (0.001–0.01 %). Допуск ±10 % — это **в 1000+ раз** шире физически возможного. При нестабильной сети регрессия скорости (`updateServerTiming`, `index.html:1939`) на шумных сэмплах может уехать к границе. Темп 1.10 = отсчёт идёт на 10 % быстрее реального; два экрана с `rate` 1.05 и 0.97 разойдутся на ~80 мс за секунду, и за минуту это уже секунды расхождения и слышимый разнобой сигналов.

Сглаживание и дедбенд снижают вероятность, но **сам предохранитель задан некорректно** — он не предохраняет.

**Фикс (суть).** Сузить кламп до реалистичного, например `0.999…1.001` (±0.1 %, с запасом). Дополнительно — отбраковывать оценку скорости при недостаточном «рычаге» (короткий `modelSpan`) и держать `rate=1`, пока нет уверенной длинной серии. См. [план, шаг 2.1](04-improvement-plan.md#s21).

<a id="b4"></a>
## B4. «Молчаливый» пропуск сигнала у границы сегмента (High, цель А)

**Где.** `index.html:3199`, `:3465`, `:3501`

```js
function isServerSignalCurrent(targetServerTime, graceMs = staleSignalGraceMs) {
  if (!targetServerTime) return true;
  return serverNow() - targetServerTime <= graceMs;   // staleSignalGraceMs = 500
}
...
if (!isServerSignalCurrent(targetServerTime)) return;  // колбэк НЕ вызывается
```

**Проблема.** Если перевзводящийся таймер проснулся позже цели более чем на 500 мс (сборка мусора, кратковременный троттлинг вкладки, рывок на слабом устройстве, восстановление из сна на грани окна), сигнал **просто не воспроизводится** — ни звук, ни вспышка. Для соревновательного таймера пропущенный сигнал «старт»/«конец» — серьёзный сбой, хуже, чем сигнал с небольшим опозданием.

У переходов фаз есть отдельный fallback (`schedulePhaseTransitionAt`), но он сам вызывается через `scheduleServerTimeoutAt` с тем же guard'ом — то есть при опоздании > 500 мс fallback тоже подавляется.

**Фикс (суть).** Различать «сигнал устарел навсегда» и «опоздал, но всё ещё в текущем сегменте». Если событие принадлежит **текущему активному** сегменту/фазе и ещё не проигрывалось — воспроизвести немедленно (хотя бы вспышку + короткий звук), а не глушить. Окно grace для «выкидывания» оставить только при реально сменившемся сегменте. См. [план, шаг 2.2](04-improvement-plan.md#s22).

<a id="b5"></a>
## B5. Накопление дрейфа во время офлайна (High, цель А+Б)

**Где.** `serverNow()` (`index.html:1851`) во время разрыва продолжает экстраполировать с последним `serverClockRate`.

**Проблема.** При продолжении отсчёта в офлайне (это правильное поведение!) модель часов больше не корректируется. Если на момент обрыва `rate` был оценён с ошибкой (а из-за B3 он может быть заметно ≠ 1), экран накапливает дрейф ~`|rate-1|` за секунду. За 30-секундный обрыв при rate=1.02 это +0.6 с. При восстановлении коррекция приходит, но кламп 250 мс/обновление растягивает сходимость → виден скачок/доводка и временное расхождение экранов.

**Фикс (суть).** На время офлайна **замораживать** `rate` в 1.0 (доверять только зафиксированному офсету), а оценку скорости применять лишь при активной свежей серии сэмплов. Это снижает дрейф во время разрыва. Связано с B3. См. [план, шаг 2.1](04-improvement-plan.md#s21).

<a id="b6"></a>
## B6. Потеря управляющих действий без обратной связи (Medium, цель Б)

**Где.** `index.html:2486` (`sendServerAction`) и `index.html:3823`–`3848` (`startTimer`).

```js
const sent = await sendServerAction("start", { ... });
if (sent) return;
if (!serverAvailable) return;   // ← при сбое markServerFailure(true) уже сделал serverAvailable=false
```

**Проблема.** `sendServerAction` при ошибке вызывает `markServerFailure(true)` → `serverAvailable=false` немедленно. После этого ветка «оптимистичного локального старта» (3833+) недостижима, и `startTimer` тихо выходит. Для оператора это выглядит как «нажал Старт — ничего не произошло», без сообщения и без повтора. При кратком сетевом глитче команда **теряется**.

**Фикс (суть).** Очередь команд с идемпотентным повтором (по `clientId`+seq) и явная индикация «команда отправляется/не доставлена / повторяю». Для primary — продолжать локально и до-синхронизировать сервер при восстановлении. См. [план, шаг 3.1](04-improvement-plan.md#s31).

<a id="b7"></a>
## B7. Единая точка отказа, нет автоперезапуска (Medium, цель Б)

**Проблема.** Вся модель завязана на один процесс на одном компьютере. Падение процесса, сон ОС, закрытие лаунчера = остановка соревнования. Нет супервизора процесса и нет механизма продвижения дисплея в сервер.

**Фикс (суть).**
- Минимум: обернуть Node супервизором (рестарт при падении) + persistence из B2 → быстрое самовосстановление.
- Опционально: подавлять засыпание ОС на сервере, предупреждать оператора.
- Долгосрочно: резервный сервер/выбор лидера (сложно, вне MVP).

См. [план, шаг 3.2](04-improvement-plan.md#s32).

<a id="b8"></a>
## B8. Нет аутентификации управления (Medium)

**Где.** `serve-bouldering-timer.js:634` — `POST /api/action` принимает `reset`/`pause`/`start`/`primary` от любого клиента в сети. Роль «primary» — только UI-договорённость.

**Проблема.** Любое устройство в LAN (по ошибке или намеренно) может сбросить/перезапустить идущий соревновательный таймер или перехватить роль primary. Для официальных стартов — риск целостности.

**Фикс (суть).** Простой PIN/токен управления (вводится на пульте, хранится в `params.txt`/сессии), проверяется на сервере для мутирующих действий; чтение/отображение остаётся открытым. См. [план, шаг 3.3](04-improvement-plan.md#s33).

<a id="b9"></a>
## B9. Возможное «зависание» SSE без `onerror` (Medium, цель Б)

**Где.** `index.html:2596` — реакция только на `eventSource.onerror`. Сервер шлёт `ping` каждые 15 с (`serve-bouldering-timer.js:618`), но клиент его не использует как heartbeat.

**Проблема.** При «тихой» смерти соединения (NAT/firewall timeout, прокси-буферизация) `onerror` может не сработать; push перестаёт приходить, а 2-секундный поллинг продолжает работать (это спасает состояние), но мгновенность теряется и переключение фаз может запаздывать до 2 с на конкретном экране.

**Фикс (суть).** На клиенте отслеживать время последнего SSE-события (включая `ping`); если тишина > ~20 с — принудительно `restartEventStream()`. См. [план, шаг 3.4](04-improvement-plan.md#s34).

<a id="b10"></a>
## B10. Нет автоматических тестов логики времени (Medium, цель А)

**Проблема.** Ядро ценности — корректность времени, но нет ни одного теста на: медиану/офсет/регрессию скорости (`updateServerTiming`), границы сегментов (`getCurrentSegment`/`buildTimeline`), `scheduledStartTime` (старт в прошлом + `restorePast`, переход через полночь), `finalizeOneShot`/`finalizeScheduledCountdown`. Любая правка синхронизации рискует регрессией, которую заметят только на реальном старте.

**Фикс (суть).** Выделить чистые функции времени в тестируемый модуль и покрыть юнит-тестами с инъекцией виртуальных часов; добавить симуляцию «скачок сервера / дрожащая сеть». См. [план, шаг 4.1](04-improvement-plan.md#s41).

<a id="b11"></a>
## B11. Рост state-пейлоада с числом клиентов (Low)

**Где.** `publicState` (`serve-bouldering-timer.js:523`) включает полный `publicClients()` со всей диагностикой; рассылается всем по SSE на каждое изменение и отдаётся каждому поллу 2×/с.

**Проблема.** При десятках экранов полезная нагрузка и частота создают квадратичную нагрузку (каждый видит всех). На ожидаемом масштабе LAN (единицы–десятки) это терпимо, но не масштабируется и раздувает трафик на слабом Wi-Fi (что косвенно бьёт по B5).

**Фикс (суть).** Разделить «горячее» состояние таймера (маленькое, для рассылки) и «холодную» диагностику клиентов (по запросу/реже). См. [план, шаг 4.2](04-improvement-plan.md#s42).

<a id="b12"></a>
## B12. Дрейф AudioContext vs performance.now() (Low, цель А)

**Где.** Планирование буфера: `delayUntilServerTime` (по `performance.now()`) → `beep(kind, delay, true)` → `AudioContext.currentTime + delay`. Это два независимых генератора часов.

**Проблема.** На горизонте до `audioScheduleAheadSeconds = 2.8` с расхождение пренебрежимо (<1 мс), так что сейчас не критично. Но при увеличении горизонта планирования или на устройствах с заметным дрейфом аудио-часов возможен сдвиг звука.

**Фикс (суть).** Периодически измерять смещение `AudioContext.currentTime ↔ performance.now()` и учитывать при расчёте `delay`. Низкий приоритет. См. [план, шаг 4.3](04-improvement-plan.md#s43).

<a id="b13"></a>
## B13. Троттлинг фоновой вкладки (Low, цель А)

**Проблема.** Если вкладка дисплея уходит в фон (системный оверлей, переключение приложения на ТВ/планшете), `requestAnimationFrame` останавливается, `setTimeout` троттлится до ≥1 с — сигналы могут опаздывать/пропускаться (усиливает B4). Wake Lock держит экран, но не гарантирует foreground вкладки. `resumeClientSync` чинит при возврате.

**Фикс (суть).** Документировать жёсткое требование «вкладка дисплея на переднем плане»; при возможности — резервный путь воспроизведения через заранее запланированные узлы AudioContext (которые играют даже при троттлинге JS). См. [план, шаг 4.4](04-improvement-plan.md#s44).
