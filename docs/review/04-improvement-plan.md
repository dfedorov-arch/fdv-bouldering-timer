# 04. Пошаговый план улучшений

План разбит на фазы по убыванию критичности. Каждый шаг: цель, что сделать, критерий приёмки. Привязка к рискам — в [03-bottlenecks.md](03-bottlenecks.md).

Принцип: сначала закрыть два требования-якоря — **(А) точная синхронизация** и **(Б) сеть/сбой не останавливают таймер**, и только потом удобства.

---

## Фаза 1 — Критично (корректность и надёжность ядра)

<a id="s11"></a>
### Шаг 1.1. Перевести серверный отсчёт на монотонные часы — устраняет [B1](03-bottlenecks.md#b1)

**Что сделать.**
1. В `serve-bouldering-timer.js` ввести монотонный источник: `const monoNow = () => Number(process.hrtime.bigint() / 1000000n)` (мс) или `performance.now()` из `perf_hooks`.
2. Хранить старт раунда как монотонную метку `startedAtMono` и считать `elapsedSeconds` по ней.
3. Дополнительно хранить настенную `startedAtWall` (на базе `Date.now()`) **только** для отображения «время старта» и для планового старта `scheduledStartTime`.
4. Для синхронизации офсета клиент по-прежнему получает `now`/`serverSentAt`/`serverReceivedAt` на `Date.now()` — это корректно (клиент маппит свои настенные часы на серверные). Но `elapsed` и `serverStartedAt`, передаваемые клиенту, должны вытекать из монотонного отсчёта, переведённого в текущую серверную шкалу `now`.

**Тонкость.** Клиентский `serverStartedAt` сейчас = `timerState.startedAt` (настенное). Нужно отдавать клиенту `serverStartedAt = now - elapsed*1000`, где `elapsed` посчитан монотонно — тогда скачок настенных часов сервера не сдвинет раунд.

**Критерий приёмки.** Ручной перевод системных часов сервера на ±30 с во время идущего раунда **не меняет** отображаемое оставшееся время на экранах (после очередной синхронизации остаётся прежним).

<a id="s12"></a>
### Шаг 1.2. Персистентность и восстановление состояния — устраняет [B2](03-bottlenecks.md#b2)

**Что сделать.**
1. При каждом `version++` сохранять `timerState` в файл (например `state.json`) атомарно: запись во временный файл + `fs.renameSync`. Дебаунс ~250 мс, чтобы не молотить диск.
2. Хранить старт как абсолютную метку (настенную для восстановления + признак «running»).
3. На старте сервера — читать файл; если был `running`, пересчитать `startedAt`/`startedAtMono` так, чтобы `elapsed` соответствовал прошедшему реальному времени (с разумным потолком, например не восстанавливать раунды старше N часов).
4. Логировать факт восстановления.

**Критерий приёмки.** Убийство процесса Node во время идущего раунда и повторный запуск → раунд продолжается с корректного `elapsed`, экраны при переподключении показывают правильное время.

<a id="s13"></a>
### Шаг 1.3. Авто-рестарт серверного процесса — частично закрывает [B7](03-bottlenecks.md#b7)

**Что сделать.**
1. В лаунчерах (`launcher/unix`, `launcher/windows`) или в обёртке запуска перезапускать Node при неожиданном завершении (с backoff).
2. На сервере подавлять системный сон во время `running` (где доступно) либо явно предупреждать оператора в UI.

**Критерий приёмки.** Принудительное завершение Node приводит к автоподъёму < 3 с; совместно с 1.2 раунд не теряется.

## Фаза 2 — Высокий приоритет (точность синхронизации и сигналов)

<a id="s21"></a>
### Шаг 2.1. Реалистичный предел скорости часов + заморозка в офлайне — устраняет [B3](03-bottlenecks.md#b3), [B5](03-bottlenecks.md#b5)

**Что сделать.**
1. Сузить кламп: `syncRateMin = 0.999`, `syncRateMax = 1.001` (±0.1 %, физически с большим запасом). При желании ещё уже.
2. Применять оценку `rate` только при «здоровой» серии: длинный `modelSpan`, низкий джиттер, достаточно сэмплов; иначе держать `rate = 1`.
3. При `serverAvailable === false` (офлайн) принудительно `serverClockRate = 1` и не обновлять `rate`, доверяя только зафиксированному офсету — меньше дрейфа за время разрыва.

**Критерий приёмки.** На стенде из 3+ браузеров при искусственном джиттере сети расхождение `elapsed` между экранами не превышает ~50 мс; за 30-секундный офлайн дрейф не превышает ~100 мс.

<a id="s22"></a>
### Шаг 2.2. Убрать «молчаливый» пропуск сигналов — устраняет [B4](03-bottlenecks.md#b4)

**Что сделать.**
1. В `scheduleSegmentTimeoutAt`/`scheduleServerTimeoutAt` различать два случая опоздания:
   - событие относится к **текущему** активному сегменту/фазе и ещё не отыграно → **проиграть немедленно** (вспышка + звук), пометить ключ как исполненный;
   - сегмент уже сменился (событие неактуально) → пропустить.
2. `isServerSignalCurrent` использовать только для второго случая (реальная смена сегмента), а не как глобальный глушитель по 500 мс.
3. Гарантировать, что fallback перехода фазы не подавляется тем же guard'ом.

**Критерий приёмки.** Искусственная пауза JS (например, `debugger`/нагрузка) на 1–2 с прямо перед границей сегмента не приводит к «немому» переходу — сигнал и вспышка отрабатывают (пусть с опозданием), переход фиксируется один раз.

<a id="s23"></a>
### Шаг 2.3. Логирование качества синхронизации для разбора инцидентов

**Что сделать.** На сервере собирать и (опционально) писать в лог сводку `syncQuality` по клиентам (offset/error/jitter/rate), плюс события «офлайн/восстановление». Это даёт фактуру для проверки целей А/Б на реальных стартах.

**Критерий приёмки.** После старта доступен лог, по которому видно макс. расхождение и эпизоды деградации сети.

## Фаза 3 — Средний приоритет (устойчивость операций и безопасность)

<a id="s31"></a>
### Шаг 3.1. Очередь и повтор управляющих команд + индикация — устраняет [B6](03-bottlenecks.md#b6)

**Что сделать.**
1. Команды снабдить `seq` и слать с идемпотентным повтором; сервер игнорирует дубликаты по `(clientId, seq)`.
2. UI-индикация состояния доставки: «отправляется / доставлено / повтор / не доставлено».
3. Для primary при сбое — продолжать локально (как в standalone) и до-применять на сервере при восстановлении вместо тихого no-op.

**Критерий приёмки.** Кратковременный обрыв сети (1–3 с) в момент нажатия «Старт/Пауза» не теряет команду: она доезжает после восстановления, оператор видит статус.

<a id="s32"></a>
### Шаг 3.2. Усиление отказоустойчивости сервера — расширяет [B7](03-bottlenecks.md#b7)

**Что сделать.** Поверх 1.3: предупреждение оператору, если сервер-компьютер на батарее/уходит в сон; health-индикатор сервера в UI. (Резервный сервер/лидер — отдельной большой задачей, вне текущего плана.)

**Критерий приёмки.** Оператор заранее видит риск (сон/батарея); health сервера отображается.

<a id="s33"></a>
### Шаг 3.3. PIN/токен на управляющие действия — устраняет [B8](03-bottlenecks.md#b8)

**Что сделать.**
1. Опциональный `controlPin` в `params.txt`.
2. Сервер требует токен для мутирующих `POST /api/action` (start/pause/reset/primary/settings...), чтение/SSE — открыты.
3. Пульт вводит PIN один раз, хранит в сессии.

**Критерий приёмки.** При включённом PIN устройство без токена не может сбросить/перезапустить таймер (получает 401/403), дисплеи продолжают отображать.

<a id="s34"></a>
### Шаг 3.4. Heartbeat-детекция SSE на клиенте — устраняет [B9](03-bottlenecks.md#b9)

**Что сделать.** Запоминать время последнего любого SSE-события (`state`/`ping`); если тишина > ~20 с — `restartEventStream()`. Учитывать `ping` сервера (он уже шлётся каждые 15 с).

**Критерий приёмки.** Имитация «тихой» смерти SSE (без `onerror`) приводит к авто-переподключению в течение ~20 с; push-обновления возобновляются.

## Фаза 4 — Качество и долгосрочное (Low/Medium)

<a id="s41"></a>
### Шаг 4.1. Юнит-тесты логики времени — устраняет [B10](03-bottlenecks.md#b10)

**Что сделать.**
1. Вынести чистые функции в тестируемый модуль: `getCurrentSegment`/`buildTimeline`, `scheduledStartTime`, `finalizeOneShot`/`finalizeScheduledCountdown`, статистика синхронизации (`median`, оценка офсета/скорости).
2. Тесты с виртуальными часами и сценариями: старт в прошлом + `restorePast`, переход через полночь, дрожащая сеть, скачок сервера, длинные/нулевые отдыхи, oneShot-границы.
3. Подключить лёгкий тест-раннер (встроенный `node:test`, без зависимостей).

**Критерий приёмки.** `node --test` зелёный; критические граничные случаи покрыты.

<a id="s42"></a>
### Шаг 4.2. Разделение «горячего» состояния и диагностики — устраняет [B11](03-bottlenecks.md#b11)

**Что сделать.** В SSE/часто-поллинге слать минимальный объект таймера; полный список клиентов с диагностикой отдавать по отдельному запросу/реже (например, раз в 5–10 с или по требованию панели).

**Критерий приёмки.** Размер часто-рассылаемого пейлоада не растёт с числом клиентов; на 20+ экранах трафик стабильно низкий.

<a id="s43"></a>
### Шаг 4.3. Учёт дрейфа AudioContext ↔ performance.now() — устраняет [B12](03-bottlenecks.md#b12)

**Что сделать.** Периодически измерять смещение между `AudioContext.currentTime` и `performance.now()`, учитывать при расчёте `delay` для буферного звука. Низкий приоритет (сейчас горизонт мал).

**Критерий приёмки.** При увеличенном горизонте планирования звук не уходит относительно вспышки/других экранов.

<a id="s44"></a>
### Шаг 4.4. Стойкость к троттлингу вкладки — устраняет [B13](03-bottlenecks.md#b13)

**Что сделать.** Документировать требование «вкладка дисплея на переднем плане + Wake Lock»; где возможно — заранее планировать звуковые узлы в AudioContext (играют даже при заморозке JS), чтобы критичные сигналы отрабатывали при кратком троттлинге.

**Критерий приёмки.** При кратком уходе вкладки в фон у границы сегмента сигнал всё равно звучит (через заранее запланированный аудио-узел).
