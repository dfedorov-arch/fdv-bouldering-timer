## TL;DR

Реализовация проекта: сервер — единый источник истины, клиенты строят собственную модель серверных часов поверх `performance.now()` (медиана офсета по «лучшим» сэмплам + оценка дрейфа линейной регрессией), сигналы планируются по серверному времени с дозапуском и упреждающей загрузкой звука. Дисплеи продолжают отсчёт во время кратких разрывов связи.

Главные риски для заявленных целей:

1. **Сервер считает время по** `Date.now()` **(настенные часы).** Скачок системного времени сервера (NTP-степ, ручная правка) сдвигает отсчёт сразу на всех экранах. → **Critical**, [03](03-bottlenecks.md#b1).
2. **Состояние таймера хранится только в памяти.** Перезапуск/падение серверного процесса = полная потеря идущего отсчёта, восстановления нет. Это худший случай для «сеть/сбой не должны останавливать таймер». → **Critical**, [03](03-bottlenecks.md#b2).
3. **Допуск на оценку «скорости часов» ±10 %.** Слишком широкий предел позволяет шуму сети раскачивать темп клиента → расхождение секунд и звуковых сигналов между экранами. → **High**, [03](03-bottlenecks.md#b3).
4. **Сигнал может быть «молча» пропущен** при подвисании вкладки/GC рядом с границей сегмента (окно `staleSignalGraceMs = 500 мс`). → **High**, [03](03-bottlenecks.md#b4).
5. **Управляющие действия теряются без обратной связи** при кратком сбое сети (тихий no-op вместо очереди/повтора). → **Medium**, [03](03-bottlenecks.md#b6).
6. **Единая точка отказа + нет аутентификации:** любой в LAN может сбросить/перезапустить соревновательный таймер. → **Medium**, [03](03-bottlenecks.md#b7), [03](03-bottlenecks.md#b8).
